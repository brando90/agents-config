# INDEX_RULES.md — Global Rules & Doc Routing

Load only the docs relevant to your current task.

**Remote fallback:** If `~/agents-config/` is not available locally (e.g., running in a cloud sandbox or on the phone), fetch any needed files from `https://raw.githubusercontent.com/brando90/agents-config/main/`. This applies to this file, all machine docs, all workflow docs, and anything else referenced below.

## Hard Rules (every response, never skip)

These rules apply to EVERY agent response and EVERY session. Violating any of these is a failure.

1. **Never commit secrets — default is NEVER, even when explicitly asked.** Before any commit (and before any auto-push backstop), scan staged files for: API keys, tokens, passwords, `.env*` (including `.env.local`, `.env.prod`, etc.), `*.pem`, `*.key`, `id_rsa*`, `credentials*`, anything under `~/keys/`, and high-entropy values assigned to names matching `(api[_-]?key|token|password|secret|auth|bearer)\s*[:=]\s*["']?[A-Za-z0-9_\-/+]{16,}`. If anything matches, abort the commit and tell the user exactly which file/line. Once a secret is pushed it must be assumed compromised — git history can't be fully purged (history rewrite + force-push + every existing clone keeps re-leaking), so rotate the credential immediately rather than trying to "scrub" it. **Even an explicit user override ("just commit it") should get pushback first**; the user has stated they don't want to be saying that, so treat such requests as a likely mistake and confirm twice before honoring, then still rotate the credential after. Use env vars or reference `~/keys/` / `~/.ssh/config` instead of inlining secrets.
2. **Verify before pushing.** Review diffs for secrets, unintended changes, broken imports.
3. **Run QA before reporting any non-trivial task as done.** Design: **A1 builds → A2 does full QA.** Dispatch one independent reviewer for full QA (correctness + structural in one pass). The reviewer finds AND fixes issues. Fallback chain: primary cross-agent reviewer → Gemini CLI → self-review with best model. One QA dispatch per user-assigned task, not per commit. Structural checks are skipped for markdown-only repos. When unsure whether to run QA, run it. **Quick dispatch:** `codex exec --full-auto -m gpt-5.5 -c 'model_reasoning_effort="xhigh"' "$QA_PROMPT"` (if CC built) or `clauded -p "$QA_PROMPT"` (if Codex built). **CLI-only, no API keys.** QA must go through the locally-logged-in CLIs (`codex`, `clauded`, `gemini`) using their cached subscription/OAuth credentials. Never fall back to `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY` — if a CLI isn't authed, skip that stage, note it in the report, and tell the user to re-run the CLI's interactive login. See [`workflows/qa-correctness.md`](workflows/qa-correctness.md) for the full protocol and [`workflows/qa-structural.md`](workflows/qa-structural.md) for structural check details.
4. **Dual TLDR: one at the top, one at the end.** Every response must open with a `**TLDR-start:**` line and end with a `**TLDR-end:**` line — 1–2 sentences each. The top TLDR exists so the user sees a summary in preview / prefix-s mode without scrolling. The bottom TLDR is the authoritative one: it must be written last, from the actual response content, **ignoring what was written in `TLDR-start`**. Do not copy-paste `TLDR-start` into `TLDR-end`. If the chain-of-thought in the response changed your conclusion, `TLDR-end` should reflect that change — divergence between the two is expected and fine. If only one TLDR is present, it must be `TLDR-end` (never only `TLDR-start`). Applies to all agents (Claude Code, Codex, etc.). No exceptions.
5. **Verification snapshot after TLDR-end.** Immediately after `TLDR-end`, append a `**Snapshot:**` block — a small, concrete, inspectable sample of the actual artifacts the task produced, so the user can visually confirm the work was really done without re-running commands. Target ~5–15 lines total; hard cap 25. Pick the *smallest representative sample of real output*, not a restatement of the TLDR and not an abstract description. Examples: 3–5 rows of a generated table/dict, first/last lines of a written file plus its path and size, URLs of pushed commits/PRs, the diff's `--stat` summary, a directory listing of new files, the exact command executed and its first few lines of output. For intangible tasks (a decision, explanation, refusal) include a one-line before/after or a cited file path + line range. If there is genuinely nothing to show, write `**Snapshot:** _(nothing to show — [brief reason])_` — do not silently omit. Applies to all agents.
6. **Refresh agents-config before each new user task.** At the start of every non-trivial user request: (1) `git -C ~/agents-config pull` to fetch remote changes, (2) re-read `~/agents-config/INDEX_RULES.md` into your active context so the current session has the latest rules and doc references, (3) if the pull brought changes, also re-read any changed machine or workflow files relevant to your current environment. For long sessions (over 1 hour), also refresh mid-task — check with `date` and compare to your last pull. Push any uncommitted agents-config changes and run QA if changes were pulled from another agent.
7. **Agent CLI freshness.** Claude Code's `SessionStart` hook (`~/.claude/settings.json`) runs `~/agents-config/scripts/auto-update-tools.sh` on every session start. Major CLIs: `codex`, `claude` / `clauded`, `gemini`. Example extras: Cursor, Mistral, Harmonic, Axiom, and similar installed tools. If one looks stale, use the right package manager or print a one-line reminder with the exact update command. Never assume `npm` for a Homebrew-installed binary.
8. **Codex defaults to GPT-5.5 xhigh.** Codex CLI should always default to `gpt-5.5` with `model_reasoning_effort = "xhigh"`. Keep `~/.codex/config.toml` set with `model = "gpt-5.5"` and `model_reasoning_effort = "xhigh"`. When dispatching Codex from another agent, pass the settings explicitly: `codex exec --full-auto -m gpt-5.5 -c 'model_reasoning_effort="xhigh"' "$QA_PROMPT"`.
9. **CLI-only for all LLM work — no agent-authored direct API calls.** Extends the CLI-only mandate from Hard Rule 3 (QA dispatch) to **every** LLM-driven action in every session. **Allowed:** dispatch via the locally-authenticated CLIs `clauded` / `codex exec` / `gemini`, which use cached subscription / OAuth credentials, are billed against the user's CLI plan with effective per-session ceilings, and leave observable transcripts under `~/.claude/` / `~/.codex/` / `~/.gemini/`. **Forbidden:** authoring or executing any direct LLM-provider API call (`anthropic.messages.create`, `openai.chat.completions.create`, `litellm.completion`, `google.genai.Client(...)`, raw `requests.post` to `api.anthropic.com` / `api.openai.com` / `generativelanguage.googleapis.com`, etc.) unless **Brando wrote the call himself**. An agent must never *author* such code — not in a new script, not in a refactor, not in a one-off notebook cell. Before *executing* any existing script that loads `~/keys/anthropic_*` / `~/keys/openai_*` / `~/keys/gemini_*` / `~/keys/aristotle_*` or sets `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `GEMINI_API_KEY`, the agent MUST pause, surface the script path + estimated spend + which experiments will run, and wait for explicit confirmation — silent runs are the failure mode. **Why:** API keys bill per-token with no cap. One agent-driven default (Opus 4.7 in an agent loop, no Batch API discount, ~30 callers all defaulting to the same key) burned **$17,752.98 in 25 days** against `anthropic_bm_key_koyejolab` in May 2026, and the lab confirmed on 2026-05-25 that it cannot absorb that scale of spend. CLI subscriptions self-throttle when a loop misbehaves; raw API self-bills until it drains the account. **How to apply:** (a) if you are about to write `Anthropic(...)` / `OpenAI(...)` / `genai.Client(...)` / `litellm.completion(...)` or paste an `sk-ant-` / `sk-` key into a script, stop and ask Brando whether to use `clauded -p "$PROMPT"` / `codex exec --full-auto -m gpt-5.5 -c 'model_reasoning_effort="xhigh"' "$PROMPT"` instead; (b) when reviewing diffs or QA'ing code, flag any agent-authored direct API call as a CRITICAL issue blocking merge; (c) before running any script that reads from `~/keys/*` for an LLM provider, surface path + back-of-envelope cost (`tasks × calls × ~$0.003 Sonnet input / ~$0.015 Opus input` per 1K tokens, multiplied by ~1–10 reps) and wait. **Exception:** Brando-authored API-calling code (commit author = Brando, or explicit "I wrote this, run it now" instruction from Brando in the current prompt) — you may execute it, but still surface estimated spend first. **Counterexample (NOT a direct API call):** invocations of the `clauded` / `codex` / `gemini` CLIs are exempt, even though they internally reach Anthropic / OpenAI / Google — they are subscription-billed and observable in local transcripts.

## Trigger Rules (mandatory when triggered)

These rules fire in specific contexts. When the trigger condition is met, they are mandatory.

6. **Commit, push, and re-read agents-config after any edit.** _Trigger: you or the user modify any file under `~/agents-config/`._ (1) Re-read the changed file(s) so your context stays current for the rest of the conversation. (2) Commit and push to the remote before continuing with the next task — other agents on other machines pull from this repo, so stale config causes drift. Also ensure project-level config files that reference agents-config (e.g., a project's `CLAUDE.md`) stay consistent.
7. **Keep PRs short.** _Trigger: creating a pull request._ The PR description must start with a **Summary** section of 5–10 bullet points max (one line each). Follow with a short **Test plan** (2–3 bullets). Include links (W&B Report URLs, relevant docs). After these two sections, an optional **Appendix** may contain extended details, file lists, or context — but assume the reader stops after the summary. No walls of text.
8. **Commit and push after QA passes.** _Trigger: QA verdict is PASS or FIXED with 0 critical issues._ Immediately `git commit` and `git push` without asking the user. Only escalate to the human if the QA verdict includes critical issues (CRITICAL_ISSUES > 0) or is FAIL. Major-only issues in non-active files (archives, historical results) do not block the commit.
9. **GPU discipline: baseline → estimate → ask → verify → clean up.** _Trigger: about to launch a GPU job._ Before launching any experiment on a GPU, first run or reason from a 1-device baseline whenever practical. Report to the user: (a) estimated VRAM, (b) expected utilization pattern (compute-bound vs CPU/IO-bound), and (c) estimated duration, or explicitly state that the duration is unknown. If the workload appears mostly CPU or IO bound, or uses only brief compute bursts, say so clearly and suggest a smaller machine or a CPU-only node. Do not silently claim multiple devices. Present the per-device allocation plan and wait for user approval. Hardware allows sharing, but PyTorch's caching allocator means memory footprints can spike unpredictably, causing OOM crashes for others. Furthermore, spreading low-utilization work across multiple devices hogs the system's PCIe bandwidth. Therefore, the etiquette rule is: a small memory footprint on one shared device is fine; spreading low-utilization work across multiple devices, or holding idle devices for long durations, must be avoided. **Hard Rule:** The agent MUST explicitly set `export CUDA_VISIBLE_DEVICES=<chosen_ids>` in the bash environment before executing any python script. Verify with `nvidia-smi` within a few minutes of launch. If utilization stays persistently low, report it to the user and suggest consolidating, fixing the input pipeline, or moving off the device. After completion, clean up orphaned processes and report the final state. See [`workflows/expts-and-results.md`](workflows/expts-and-results.md) § Allocation Rules and § Post-Experiment Cleanup.
10. **Mega QA: sequential multi-model review chain.** _Trigger: user says "mega QA", "super QA", "extra careful QA", "deep QA", or similar._ Instead of the default single-reviewer QA, run all 3 models **sequentially** in 3 stages — each does full QA (correctness + structural, with authority to fix), then passes the improved code to the next. Default chain (3 stages, one per model): Codex → CC → Gemini (if CC built) or CC → Codex → Gemini (if Codex built). The builder reviews in the middle (it knows the intent best), Gemini always does the final clean-eyes pass. If the user asks for more rounds, cycle through the chain again (e.g., 6 rounds = chain × 2). **Per-stage fallback (never bail):** if any stage fails (auth, credits, context overflow, 5xx, sandbox blocks), coin-flip among the other working CLIs and otherwise self-dispatch the driving agent — always complete the full N stages, even if the worst case is `CC → CC → CC`. Use case: end of work day, before major merges, before sleep — when you can afford the compute for a thorough review. See [`workflows/qa-correctness.md`](workflows/qa-correctness.md) § Mega QA.
11. **Publish ultimate-utils to PyPI after QA-gated push to `master`.** _Trigger: QA passes on `~/ultimate-utils/` and changes are pushed to `master`._ After the QA-approved code is on `master`: (1) bump the `version` field in `~/ultimate-utils/pyproject.toml` (patch increment, e.g., `0.10.3` → `0.10.4`), (2) commit and push the version bump to `master`, (3) build and upload: `cd ~/ultimate-utils && bash scripts/publish_to_pypi.sh` (handles token auth from `~/keys/push-pypi-all.txt`, builds, checks, and uploads). This keeps Git and PyPI in sync so other machines and downstream users get the latest via `pip install ultimate-utils --upgrade`.
12. **Clean up idle background processes (>24h, confirmed idle).** _Trigger: starting a new session, or starting new monitors/workers._ Before spawning new background processes, scan for existing ones (`ps aux | grep -E "(tail -f|run_worker|until.*sleep.*done)" | grep -v grep`). For each candidate, run a health check before killing — **never kill blindly**: (1) CPU time delta — sample `ps -p $PID -o cputime=` twice 10s apart; if cputime increased, it's actively computing, don't kill (single-sample `%cpu` is unreliable between bursts). (2) Has child processes (`pgrep -P $PID`)? → don't kill. (3) For workers: DB has claimed jobs? → don't kill. (4) For `tail -f`: log file modified in last hour? → don't kill. (5) For `until` loops: wait condition already met? → stuck, safe to kill. **Only kill if ALL checks confirm idle AND running >24h.** Also: name processes descriptively when spawning them so future agents can inspect purpose from `ps aux`. ScheduleWakeup loops must self-terminate when their condition is met.
13. **Load ML writing guide when working on a top-venue paper.** _Trigger: any of (a) editing `.tex` files for an ML research paper aimed at NeurIPS / ICML / ICLR / similar; (b) generating numbers, tables, or claims that will land in such a paper; (c) deciding what experimental artifacts (datasets, generations, results files) to keep, archive, delete, or cite for such a paper; (d) answering questions like "did the proofs compile?" / "how many runs passed?" / "does claim X hold up?" in the context of such a paper._ Read [`writing/ml_research/ml_research_writing.md`](writing/ml_research/ml_research_writing.md) into your context before acting. Follow its persona, abstract structure, writing rules, **§ Scientific Rigor**, and **§ Sentence-Level Prose Discipline** throughout the session — the rigor section is non-negotiable for award / test-of-time-aiming work, and the prose-discipline section is non-negotiable for any sentence that ships in a top-venue paper. Run the 4-check (read-aloud / claim-count / citation-count / word-count) at sentence-write time, not at QA review time.
14. **Email only user-triggered big/mega QA or explicitly tracked completions to Brando.** _Trigger: (a) a user-triggered "mega QA", "deep QA", "big QA", or similar substantial QA pass finishes, OR (b) Brando explicitly asked for email/notification/tracking on completion._ Send one concise email to `brando.science@gmail.com` after the final QA/commit/push state is known. Do not CC Brando's personal email on internal agent notifications unless Brando explicitly asks for it; see Trigger Rule 26. If Telegram triggered or approved the task, also send a one-line Telegram completion message in the originating chat.
    - **Default QA is not enough.** The normal QA required by Hard Rule 3 does NOT by itself trigger email. Ordinary requests should finish in chat without an extra mailbox round trip.
    - **Paper/research edits are not automatic email triggers.** Email only if the edit was part of a user-triggered big/mega QA pass or Brando explicitly requested email notification. Routine manuscript edits, table fixes, exploratory reads, and result lookups should report in chat/Telegram only.
    - **Significant by itself is not enough.** Multi-hour, long-running, or high-stakes work still does not email unless it is user-triggered big/mega QA or Brando explicitly says to notify/email/track completion. If unsure, do not email; summarize in the current chat.
15. **Q&A stays in chat unless explicitly emailed.** _Trigger: the user asks a substantive question._ Answer the question in the normal response and do not send an email by default. If Brando explicitly says to email the answer, send one email to `brando.science@gmail.com` only, with no CC unless the prompt names additional recipients. **Subject:** `Q -> A: <brief question> -> <brief answer>`. **Body:** Start with `TLDR: <1 sentence>`, then write in natural conversational prose. Skip email for ordinary clarification, diagnosis, or "why did this happen?" questions; otherwise requests become slower and every debugging conversation turns into mailbox noise.
16. **Document headers: title + 1–2 sentence TLDR on every generated doc.** _Trigger: you create a new document or substantially rewrite one (any `.md`, `.txt`, `.rst`; design docs, plans, READMEs, reports, summaries, postmortems, eval writeups)._ The first non-blank line of the file MUST be a `# Title` heading describing the doc's purpose, and immediately under it a `**TLDR:**` block of 1–2 sentences summarizing the document. Why: Brando reviews many auto-generated docs per day and cannot read every body — a missing or weak TLDR makes the work unverifiable in practice ("otherwise it's too much shit to verify"). Applies whether the doc is committed, only staged, or kept locally. Applies to docs you author or substantially rewrite, not to docs you only spot-edit. For very short docs (< ~20 lines) a one-line `**TLDR:**` is sufficient. **Distinct from Hard Rule 4:** Rule 4 is the response-level Dual TLDR for chat output; Rule 15 is the file-level Title + TLDR for documents you write to disk. Both apply when you generate a doc inside a response — the chat gets `TLDR-start`/`TLDR-end`; the file gets `# Title` + `**TLDR:**`.
17. **Run user tasks end-to-end; never ask mid-task.** _Trigger: any user task._ Flow: smoke → full E2E → QA (Hard Rule 3) → commit + `git push` (Trigger Rule 8) → only then surface to the user. If mid-task verification is needed (incl. smoke failure), delegate to Codex / Gemini / fresh `clauded -p` against the user's original prompt + `~/CLAUDE.md` — never block on the user. Halt at smoke only on explicit ask ("smoke only", "stop after smoke").
18. **Paper appendix / low-readership changes default to `main`.** _Trigger: a branch's only changes are to paper appendix files (e.g., `paper_latex/.../9*_appendix*.tex`), factual / numeric refreshes from a completed eval run (table numbers, abstract numbers consistent with appendix tables), small polishing edits, or other sections few reviewers will read closely._ Merge directly to `main` without asking for per-merge approval (use `git merge --ff` if fast-forward, otherwise a small merge commit, or fast-merged PR — whichever the repo's convention). **Why:** few people read the appendix, so the cost of waiting on explicit human approval exceeds the cost of a small revert; as long as the content is correct and high-quality, defaulting to `main` keeps the trunk fresh and avoids stale feature branches. **How to apply:** First run `git diff --stat origin/main...<branch>` and skim the actual diffs. If everything is appendix / numeric refresh / typo fix, merge to main. If the branch also touches substantive claims (abstract argument changes, intro framing, method definitions, new headline numbers in main eval) **beyond** consistency with new appendix numbers, hold and ask. The user's explicit "don't push to main" overrides this rule for that branch.
19. **Delete merged branches.** _Trigger: a feature / topic branch has been fully merged (or squashed / rebased) into `main`._ Immediately delete it both locally and on the remote: `git branch -d <name>` then `git push origin --delete <name>` — this is the standard post-merge cleanup. **Why:** keeps `git branch -a` uncluttered, removes "is this still in flight?" ambiguity, prevents accidental new commits on a finished topic. Recovery is trivial: the SHA lives on in main's merge commit (`<merge>^2`), so `git branch <name> <merge>^2` re-creates it. **How to apply:** `git branch -d` (lowercase) refuses if the branch isn't fully merged — trust that as your safety net; never escalate to `-D` without inspecting `git log <branch> --not main`. Skip for long-running release/integration branches (`release/*`, `dev`, `staging`) and shared collaboration branches that multiple agents are still pushing to.
20. **Verify Lean by actually compiling with the project's pinned toolchain — even in cloud sandboxes.** _Trigger: any request that requires Lean compilation / type-checking / proof verification (e.g., "check this proof", "does `imp ≡ functional` hold", "compile this file", "do the tests pass")._ (1) Find the pinned toolchain — `lean-toolchain` first (e.g., `~/veribench/veribench_dataset/lean_src/lean-toolchain`), else grep `.github/workflows/*.yml` for `leanprover/lean4:v` / `elan toolchain install`. (2) Install via elan and run `lake build` (or `lake env lean <file>`) for real: `curl https://elan.lean-lang.org/elan-init.sh -sSf | sh -s -- -y && export PATH="$HOME/.elan/bin:$PATH" && elan toolchain install <pinned> && elan default <pinned>`. (3) Report the actual `lake` output, not your guess. **Applies in Anthropic cloud / restricted sandboxes too** — install Lean and verify; never substitute "looks correct" for a real compile. **Why:** Brando uses Lean often; CI is the source of truth for the version that works; an unverified "this proves" claim is worse than admitting you didn't run it.
21. **Load `write-intro.md` when drafting or revising a paper introduction.** _Trigger: the user asks for help writing, drafting, or revising the **Introduction section** of a research paper (e.g., "write the intro," "draft section 1," "revise the introduction"); especially when they pass an abstract path and a paper draft path._ Read [`writing/ml_research/write-intro.md`](writing/ml_research/write-intro.md) into your context before acting. This file specifies the CS197 six-move architecture, voice, citation discipline, and compile workflow specific to introduction sections. Compose with Trigger Rule 13 (`ml_research_writing.md`) when both apply — `write-intro.md` is the section-specific scaffold; `ml_research_writing.md` is the broader writing guide.
22. **Load `write-abstract.md` when drafting or revising a paper abstract.** _Trigger: the user asks for help writing, drafting, or revising the **abstract** of a research paper (e.g., "write the abstract," "draft the abstract," "revise the abstract"); especially when they pass a title and rough ideas / free-form notes about the paper._ Read [`writing/ml_research/write-abstract.md`](writing/ml_research/write-abstract.md) into your context before acting. This file specifies the CS197 six-move scaffold compressed to ~150 words, the Sanmi / Brando quality-bar lessons (failure-mode-first opening, named metric, memorable terminology, concrete numbers, vision-not-self-congratulation closing), and the alternate-openings deliverable. Compose with Trigger Rule 13 (`ml_research_writing.md`) when both apply.
23. **CS197 jargon stays out of reader-facing prose.** _Trigger: drafting or revising any reader-facing text in a research paper, blog post, talk abstract, slide deck, or README marketing copy._ Stanford CS197 / Sanmi-lab vocabulary — `bit flip`, `the bit`, `flip the bit`, `vectoring`, `(research) velocity`, `north star`, `killer demo`, etc. — is internal planning shorthand between agents and Brando, NOT writing-style vocabulary. **Allowed:** as a `% LaTeX comment` (e.g., `% bit flip: prior assumes X, we show Y`) to guide structure, or removed entirely. **Not allowed:** in `\section{...}` titles, abstract body, intro prose, figure captions, slides, blog body, README headlines — anything a reviewer or external reader sees. **Why:** ~99% of readers haven't taken CS197 — these terms read as in-group jargon and signal an unpolished paper to top-venue reviewers; Brando has been burned by this before. **How to apply:** when drafting or revising, scan prose for these terms; either delete them, rephrase to plain scientific language ("the assumption we challenge," "our long-term vision," "the headline result"), or wrap as `%` comment. Mirrored briefly in `writing/ml_research/ml_research_writing.md`, `writing/ml_research/write-intro.md`, and `writing/ml_research/write-abstract.md` so each writing skill carries the reminder.
24. **Audience-calibrated terminology — translate subfield jargon for the reader.** _Trigger: drafting or revising reader-facing text aimed at an audience outside your subfield (e.g., a formal-methods paper read by ML reviewers, an ML paper read by HCI reviewers, a research blog post read by general technical readers)._ Subfield terms carry different connotations across audiences. Formal-methods *specification* (a logical property to prove) reads to ML reviewers as *prompt spec* or *API spec*; FM *soundness* / *completeness* are alien to most ML readers. **How to apply:** for each technical noun in the abstract / intro, ask "what would the target audience assume this means by default?" — if that's wrong, translate. *specification* → *theorem formulation* / *provable guarantee* / *universal property*; *spec generation* → *extracting a formal property to prove*. Pair with the four related antipatterns in `writing/ml_research/ml_research_writing.md` § Sentence-Level Prose Discipline (#7 audience translation; #8 bridge before escalating; #9 attach critique to fact; #10 name the missing capability, not pejoratives). **Why:** mistranslated jargon makes a reviewer dismiss a paper they didn't understand — the work isn't bad, it's just unreadable to them.
25. **Load Brando blog writing docs when drafting or revising personal blog posts.** _Trigger: the user asks to write, draft, revise, polish, make "badass," or publish a personal blog post for Brando's site; gives a rough idea dump intended to become a blog post; references `~/brandomiranda/`, the scalable oversight post, the agents-config blog post, or Brando's recent blog style._ Read [`writing/blog/rules.md`](writing/blog/rules.md), [`writing/blog/blog_writing.md`](writing/blog/blog_writing.md), and [`writing/blog/write-blog-post.md`](writing/blog/write-blog-post.md) into context before acting. If `~/brandomiranda/` is local, also skim the recent anchor posts named in `rules.md`. Use the personal-blog voice: direct first-person stance, one central question, technical rigor, memorable compression, honest uncertainty, and clean Jekyll mechanics. Compose with Trigger Rules 23-24 for reader-facing prose and terminology translation. If the target is a SAIL / lab research blog post rather than Brando's personal site, this trigger does not apply; load [`workflows/blog-posts.md`](workflows/blog-posts.md) instead. If the task is explicitly a personal adaptation of a SAIL-style post, load both and keep the target format explicit.
26. **Route Brando email without spamming personal inboxes.** _Trigger: any agent (OpenClaw triage agent, Claude Code, Codex, or any other automation) sends an email on Brando's behalf or sends an internal notification email to Brando._ Internal agent notifications to Brando (user-triggered big/mega QA completion, explicitly tracked task completion, job dispatch status when explicitly requested, OpenClaw completion receipts when explicitly tracked) MUST go to `brando.science@gmail.com` with no CC by default. Do not CC `brandojazz@gmail.com` or `brando9@stanford.edu` on internal notifications unless Brando explicitly asks. For external emails sent on Brando's behalf to another person, CC `brando.science@gmail.com` for auditability; add `brando9@stanford.edu` only when a Stanford/academic record is useful; never CC `brandojazz@gmail.com` by default. De-duplicate addresses already present in `To`. **Why:** Brando wants major agent completions tracked in the science inbox, while personal email should not receive agent-generated notification spam.
27. **Q&A leads with `**A (TLDR):**` direct one-line answer.** _Trigger: answering a substantive question from the user, in chat or in a written explainer doc (`concepts.md`, FAQ, design-decision write-up)._ Format: first content line of the answer is `**A (TLDR):** <one sentence direct answer>`. Then `**Detail.**` (or equivalent) introduces the longer explanation. The TLDR line must (a) actually answer the question, (b) fit on one line, (c) avoid hedging, (d) avoid restating the question. **Why:** Brando reads many auto-generated Q&As per day and cannot scan a wall of text to extract the answer — a missing or weak direct-answer-first makes the explainer unverifiable in practice. **Distinct from Hard Rule 4 (response-level Dual TLDR) and Trigger Rule 16 (file-level doc TLDR):** rule 4 is the chat-response TLDR, rule 16 is the file-header TLDR, rule 27 is the per-Q&A direct-answer-first format inside any explainer content. All three can fire at once on a single response that contains a Q&A inside a doc inside a chat reply. Anchored at [`experiments/01_self_hosted_openclaw/concepts.md`](experiments/01_self_hosted_openclaw/concepts.md) which establishes the format.
28. **Save user-attached images / photos / scans into the relevant experiment's `assets/` folder.** _Trigger: the user attaches an image (photo of a handwritten note, screenshot, diagram, paper figure, whiteboard) in a prompt and asks to save / transcribe / file / keep it, especially in the context of a specific experiment, repo, or project directory._ (1) Identify the target experiment / project folder from context (the user usually names it, e.g. "experiment 02_", "the openclaw experiment", or the current repo if obvious). (2) Create `<that-folder>/assets/` if it doesn't already exist (`mkdir -p`). (3) Copy the image there with a descriptive snake_case filename (`handwritten_<topic>.jpg`, `whiteboard_<date>.png`, `screenshot_<feature>.png`) — not the original opaque UUID upload name. (4) If a transcription is requested, save it as a sibling `.md` next to the image, same basename, with provenance, the original image path, and a `**TLDR:**` header per Trigger Rule 16. (5) Commit both the image and the transcription as part of normal repo workflow. **Why:** the original upload path under `~/.claude/uploads/<uuid>/` is ephemeral and opaque — it disappears across sessions and machines, and the UUID filename tells future Brando (and future agents) nothing about what's in it. Co-locating with the experiment keeps the artifact discoverable via `git log` / `ls`, lets readers cross-reference the transcription with the original handwriting / pixels, and avoids one-off "images" or "scratch" top-level folders that erode over time. **How to apply:** default to `assets/` (singular precedent: `experiments/01_toy_ebm_training/assets/`). If the user names a different subfolder (e.g. `figures/`, `notes/`), honor that instead. If the target experiment doesn't exist yet, ask — don't guess. If the image is a screenshot of a *third-party* paper / figure, also flag it in the transcription so it's clear what's original vs. cited. Mirror briefly in [`workflows/expts-and-results.md`](workflows/expts-and-results.md) so it's discoverable from the experiment workflow doc.

## Guidelines (best practices)

Follow these as conventions. They improve quality but are lower priority than Hard Rules and Trigger Rules.

14. **Just do it.** Follow direct instructions immediately. Do not draft when told to send. Do not ask for confirmation unless the action is truly destructive (e.g., force-push to main, deleting production data).
15. **Prefer references over full context loading.** Cite file paths as text (e.g., `~/agents-config/machine/mac.md`); load the file only when the task needs it. A "reference" here is a written path to a doc — not a symlink or memory address.
16. **Keep `agents-config` self-consistent.** When modifying this repo, ensure INDEX_RULES.md, README.md, and listed doc paths remain accurate.
17. **Use explicit anchored paths in prose doc references and commands.** Write `~/agents-config/INDEX_RULES.md` or `~/veribench/docs/agent-docs/INDEX.md`, never bare relative references like `docs/agent-docs/`. The user works across many repos and machines, so unanchored paths are ambiguous without context.
18. **Always use `ls -la` (not `ls`) when listing directories for keys, tokens, configs, or credentials.** Hidden (dot-prefixed) files are common for sensitive data — plain `ls` omits them. This applies to any directory likely to hold secrets (e.g., `~/keys/`, `~/.ssh/`, `~/.config/`).
19. **Always use SOTA models for experiments.** When running experiments, use the best available model from any model provider (e.g., Opus 4.6 not Sonnet 4 for Anthropic). When reporting results, always state the exact model ID (e.g., `claude-sonnet-4-20250514`). Weaker models undermine the paper's conclusions.
20. **Use Brando's email signature and voice.** When sending emails on Brando's behalf, apply the routing rule from Trigger Rule 26, append the signature from [`email-signature.md`](email-signature.md), and follow its voice rules: first person as Brando, concise, friendly, direct, with no third-person narration about Brando from Brando's own account.
21. **Check `~/keys/` for non-LLM credentials.** Before asking the user for non-LLM credentials, always check `~/keys/` first (`ls -la ~/keys/`). Common non-LLM files: `master_hf_token.txt` (HuggingFace), `brandos_wandb_key.txt` (W&B), `.brando90_github_token.txt` (GitHub), `push-pypi-all.txt` (PyPI), `gmail_app_password.txt` (Gmail SMTP). Load them with `cat ~/keys/<file>.txt` and set as env vars. **LLM-provider keys (`anthropic_*`, `openai_*`, `gemini_*`, `aristotle_*`) are excluded from this guideline** — per Hard Rule 9, agents may not load or use them; route LLM work through `clauded` / `codex` / `gemini` instead.
22. **No hardcoded Node/nvm versions in `.bashrc` PATH prepends.** `nvm.sh` (sourced in `.bashrc`) already puts the active node's bin first in PATH. A hardcoded line like `export PATH=".../.nvm/versions/node/v24.14.0/bin:$PATH"` will silently shadow a newer node after `nvm install`, causing `npm install -g <tool>` to update one install while the shell runs another — the "please update" loop bug. See [`machine/snap.md`](machine/snap.md) § "npm globals — update loop gotcha" for the diagnose/fix recipe.
23. **Email = send via uutils SMTP, MCP is fallback.** When told to send an email, use `uutils.emailing.send_email_smtp()` from `~/ultimate-utils/` (or `pip install ultimate-utils`) with `smtp_pass_file="~/keys/gmail_app_password.txt"`. This works on every SNAP node without browser auth. Only try the Gmail MCP tool if uutils is genuinely unavailable. Never create a draft when told to "send". Default from/CC per [`email-signature.md`](email-signature.md) and Trigger Rule 26: internal notifications go to `brando.science@gmail.com` with no CC; external emails CC `brando.science@gmail.com` for audit unless the user says otherwise. Contacts workflow: [`contacts.md`](contacts.md).

## SNAP watcher reauth & reboot persistence (2026-04)

DFS job-queue watchers stay alive across the two failure modes (Kerberos/AFS ticket expiry every ~10 h, and node reboot) via two cron lines on each watcher node:
```
0 */4 * * * /dfs/scratch0/brando9/bin/krenew.sh                  # keytab kinit + aklog
@reboot     /dfs/scratch0/brando9/bin/start_watcher_at_reboot.sh # wait for DFS, krenew, relaunch
```
**No password ever leaves your Mac.** The reauth credential is `/dfs/scratch0/brando9/.keytab` (chmod 600), generated once interactively via `ktutil`; `kinit -kt` proves identity from the keytab without prompting. Automation (Claude Code, cron, Codex) only needs read access to the keytab file. If you change your Stanford password, regenerate the keytab — see [`init_no_passwords_snap_kinit.md`](init_no_passwords_snap_kinit.md) and [`README.md`](README.md) § "Keeping watchers alive".

## SNAP Required Symlinks (every node)

Every SNAP node must have these symlinks in `$HOME` (LFS). Missing symlinks cause silent failures in tooling that assumes `~/...` paths resolve to DFS. `snap_setup.sh` and `machine/snap-init.md` create/verify them.

- `~/.bashrc` → `/dfs/scratch0/<user>/.bashrc`
- `~/agents-config` → `/dfs/scratch0/<user>/agents-config`
- `~/CLAUDE.md` → `~/agents-config/CLAUDE.md`
- `~/agents.md` → `~/agents-config/agents.md`
- `~/.claude` → `/dfs/scratch0/<user>/.claude` (shared auth across nodes)
- `~/keys` → `/dfs/scratch0/<user>/keys`
- **`~/dfs` → `/dfs/scratch0/<user>`** (required by the DFS job queue watcher and any tooling that uses `~/dfs/...` paths; if `~/dfs` is a plain local directory instead of the symlink, `cp file ~/dfs/job_queue/pending/` succeeds locally on LFS and is invisible to the shared queue — see `workflows/remote-job-dispatch.md` § "Is any job of mine running?")
- Plus each project dir under `~/` (e.g. `~/veribench`) as a symlink to `/dfs/scratch0/<user>/<project>`
